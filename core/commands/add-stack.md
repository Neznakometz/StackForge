# /add-stack — расширить библиотеку стеков

Ты добавляешь новый knowledge-пак, чтобы `/init` мог разворачивать проекты на
этой технологии. Пак — это правила из ОФИЦИАЛЬНЫХ доков, не из памяти.

## Шаги

1. **Уточни** технологию и её актуальную версию из аргумента `$ARGUMENTS`
   (если пусто — спроси). Определи `id` (kebab-case, напр. `frontend-svelte`)
   и `category` (backend | frontend | devops | mobile | other).
2. **Проверь дубль** в `core/knowledge/registry.json` — если пак уже есть,
   предложи `/update-knowledge <id>` вместо создания.
3. **Заземли на доки.** Найди официальную документацию, прочитай разделы
   best-practices / structure / security / testing. Бери только то, что
   подтверждено доками; помечай версию. Запрещено выдумывать правила из памяти.
4. **Создай** `core/knowledge/<id>/rules.md` и `sources.md` по шаблону из
   `core/knowledge/_template/`. Правила — операциональные и проверяемые.
   Укажи `conflictsWith`, если пак противоречит другому по мнению
   (напр. альтернативный стейт-менеджмент/ORM).
5. **Зарегистрируй** в `registry.json`: добавь объект пака
   (`status: "ready"`, `version` стартует с `1.0.0`, `last_verified` = сегодня,
   `sources` = реальные URL доков).
6. **Проверка.** `ls core/knowledge/<id>/`, перечитай rules.md на конфликты с
   соседними паками, сообщи что добавлено и как это подхватит `/init`.

## Принципы

- Один пак = одна технология (или тесная связка). Не сваливай React и Vue в один.
- Коротко: пак — правила в CLAUDE.md, не учебник.
- Конфликты обязательны к указанию — иначе `/init` склеит противоречия.
