# Дизайн API

Контрактная дисциплина (схема — single source of truth, типы из схемы) — в core-скиле `contracts`. Здесь — правила самого API.

- **Валидация на границе.** Каждый вход валидируется схемой до бизнес-логики; невалидный → 4xx с машинным кодом ошибки. Ответы тоже валидируются схемой (на обеих сторонах).
- **Ошибки типизированы.** Доменные ошибки — классы; пользователю человекочитаемое сообщение + стабильный код; в Sentry — полный контекст. Не протекай стектрейсами/внутренними деталями наружу.
- **Идемпотентность и побочные эффекты.** POST с побочкой — идемпотентный ключ там, где возможны ретраи. GET без побочных эффектов.
- **Пагинация/лимиты** на всех листингах; rate limit на чувствительных эндпоинтах.
- **Версионирование** — префикс (`/v1`); ломающее изменение = новая версия, не правка старой.
- **Безопасность** (см. `security-auditor`): authz на каждом защищённом роуте, нет IDOR (доступ по чужому id), мультитенантный скоуп (см. скил `multitenancy`).
- **Тяжёлое — в воркеры** (скил `background-jobs`): в обработчике запроса нет ASR/LLM/ffmpeg/тяжёлых вычислений.
