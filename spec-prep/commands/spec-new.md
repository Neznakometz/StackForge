# /spec-new — idea → спека

Запускается ОДИН раз в начале проекта, до кода. Цель — правильная спека как основа, чтобы потом не переделывать. НЕ пиши код.

## Шаги

1. **Brainstorm** (скил `spec-writing` → секция вопросов). Вытащи из идеи `$ARGUMENTS` спеку диалогом: для кого, какую проблему решаем, главный критерий ценности, что НЕ входит (анти-скоуп). По чанкам, давай подтверждать.
2. **Конституция.** Создай `constitution.md` по шаблону: принципы (качество, тесты, UX, перфоманс), нерушимые правила проекта, иерархия источников истины.
3. **Specify.** Заполни `SPECIFICATION.md` по шаблону: что строим и зачем, функциональные требования (FR-*), сущности/данные, API (если есть), без привязки к стеку на этом шаге.
4. **Clarify.** Пройди по недоспецифицированным местам структурированными вопросами; ответы фиксируй в секцию «Уточнения». Не уходи в план с нерешёнными конфликтами — помечай `⚠️ КОНФЛИКТ`.
5. **Design** (если есть UI). `DESIGN_SPEC.md`: токены, экраны, состояния (loading/empty/error), доступность.
6. **Plan.** `IMPLEMENTATION_PLAN.md`: разбей на фазы; каждая фаза — задачи + **проверяемые критерии приёмки** («при X происходит Y»). Это то, что потом проверяет `/phase-check`.
7. **Analyze (checklist).** Прогони спеку как «юнит-тесты для текста»: каждое требование — однозначно, измеримо, без дыр и противоречий? Прогони скил `prompt-audit` на `CLAUDE.md`, если он уже собран `/init`.
8. **Verify.** `ls` созданных файлов. Покажи владельцу summary + открытые вопросы. СТОП — дальше код начинает доменный набор (`/next-task`).

Шаблоны — в `templates/`. Не выдумывай требования: нет данных — вопрос владельцу, не догадка.
